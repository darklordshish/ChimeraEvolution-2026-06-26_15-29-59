# Конструктор-созвездия: человек — такой же донор, звезда — в слот

**Статус:** дизайн-видение пользователя (2026-07-21), spec-first перед кодом. Реализует разд. 6 GDD, уточняя его. Трогает `ConstructorUI` и модель слотов в `CreatureBody`.

## 0. Почему сейчас

Цикл по донорам — правильное решение для двух состояний и тупик для N. Замер по коду:

- **Обычный слот:** `человек → донор0 → донор1 → человек`. Два донора — три позиции, терпимо. Три — четыре, уже тяжело.
- **Химерный слот** ([CreatureBody.cs:193](../../../Assets/_Chimera/Scripts/Player/CreatureBody.cs:193)): `MakeChimeraSlot` собирает **все органы всех доноров разом** — не по типу слота, а вообще все. Три вида по ~6 органов = ~18 позиций. Это не «неудобно», это неюзабельно.
- **Человеческих органов в химерном слоте нет вообще:** `human = null`, перебираются только `donors`.

Прямой выбор (клик по звезде = поставить именно этот орган) снимает проблему структурно. Тюнингом цикла её не решить.

GDD разд. 6 уже описал целевой экран: витрувианская фигура-химера, слоты-выноски, созвездия по видам (реальные: Волк = Lupus, змея = Serpens), тяни звезду → слот подсвечивается → дроп морфит фигуру. Есть макеты в `Docs/images/`. Это не новая идея, а нереализованное решение.

## 1. Уточнение пользователя: человек — донор со своим созвездием

**Витрувианская фигура и есть созвездие человека.** Слоты — на ней, и рядом с каждым слотом его родной человеческий орган-звезда.

В GDD человек был безымянным дефолтом слота («снять звериное» = вернуться в никуда). Здесь он становится равноправным донором, просто нарисованным как тело, а не как россыпь звёзд на небе.

Что это даёт:

- **Единообразие взаимодействия.** Поставить человеческое — тот же жест, что поставить волчье. Нет особого случая «снять».
- **Химерный слот перестаёт быть исключением.** Человеческие органы доступны в нём, потому что они просто звёзды человеческого созвездия. Второе человеческое сердце — валидный графт, и это стратегия «усилиться, не звереть»: шкала мозга не дрейфует.
- **Модель упрощается в коде** (см. §3): `Slot.human` как отдельное поле исчезает.

## 2. Экран

**Центр** — фигура шасси со слотами-гнёздами; рядом с каждым гнездом родная человеческая звезда.
**По сторонам** — созвездия доноров: волк, змея, лось. Каждая звезда = орган, подписана названием и ценой со скидкой родства.
**Низ** — пул мутагена и шкала мозга (доля звериных слотов).

**Взаимодействие:**

- **Поставить:** тянешь звезду → совместимые гнёзда подсвечиваются → дроп = орган встал, пул списался. Не хватает пула — гнездо краснеет, звезда не встаёт и возвращается.
- **Снять:** тянешь родную человеческую звезду в её гнездо (или ПКМ по гнезду) → возврат к человеку, очки вернулись.
- **Применение мгновенное** по дропу, без модального «подтвердить» (как в GDD).
- **Хоткеи 1–6 остаются** — быстрый цикл по слоту для тех, кто привык; прямой выбор их не отменяет.

**Пауза — полная** (`Time.timeScale = 0`), решение пользователя. Сейчас `ConstructorUI` замедляет ×10 «чтобы чувствовать давление боя» — на практике это мешает думать над сборкой. UI уже живёт на `unscaledDeltaTime`, так что смена тривиальна.

**Арт — черновой** (решение пользователя: «функциональный главное, красоту потом»). Фигура — схематичный контур с точками-гнёздами; созвездия — группы звёзд с линиями, без реальных рисунков Lupus/Serpens. Морф фигуры по свапам, настоящие звёздные карты и красота тетради — следующим проходом.

## 3. Модель: человек как вариант

Сейчас у слота два разных места для органа:

```
Slot { Organ human;  List<Variant> variants;  int current; }   // current = -1 → человеческий
```

Отсюда особые случаи повсюду: `SlotCost` считает по-разному для `-1` и `≥0`, `NextStep` возвращает `-1` как «снять», химерный слот с `human = null` требует отдельной ветки, а пустой химерный отображается как `«—»`.

**УТОЧНЕНИЕ ПРИ РЕАЛИЗАЦИИ (2026-07-21): поле `human` НЕ исчезает — у него вторая роль.** Оно не только «что стоит по умолчанию», но и **база экспрессии**: звериный орган не заменяет человеческий, а смешивается с ним — `Blend(human, beast, m)`, где `m` — множитель родства. Низкое родство → почти человек, высокое → почти зверь, овершут → за пределом зверя. Убрать базу и считать зверя «от нуля» нельзя: на низком родстве орган выродится в пустышку и кривая раскрытия генов сломается (химерный слот именно поэтому считается от `EmptyOrgan` — там базы и правда нет).

**Целевая модель:** человеческий орган становится ещё и вариантом наравне с прочими (`species = chassis.speciesName`), а `human` остаётся ОТДЕЛЬНО — как база слота:

```
Slot { Organ human;              // БАЗА экспрессии слота (химерный: null → бленд от нуля)
       List<Variant> variants;   // включая человеческий — он выбираемый наравне с донорскими
       int current; }            // индекс варианта; -1 только для ПУСТОГО химерного
```

Что реально уходит: `current == -1` как «человеческий» (остаётся только «пусто» у химерного), ветка «снять» отдельно от «поставить», и отсутствие человеческих органов в химерном слоте.

- Скидка родства к человеку работает уже сейчас (`EffectiveCost` берёт `GetAffinity("Человек")`) — код прямо это признаёт: «человек — вид, просто ты на нём на 100 родства».
- Химерный слот наполняется теми же правилами, что обычный, плюс органы шасси → человеческие звёзды в нём появляются сами.
- `installed` (надет звериный) становится производным: `variants[current].species != chassis.speciesName`. Шкала мозга считается так же, как считалась.

**API для UI вместо цикла:**

| было | стало |
|---|---|
| `ToggleSlot(i)` — следующий по кругу | `Install(slot, variant)` / `Remove(slot)` — прямо |
| `GetSlot(i)` c `nextName`/`nextCost` | `GetSlot(i)` + `GetVariants(i)` — весь список с ценами и доступностью |
| — | `Catalog()` — звёзды по видам для созвездий |

`ToggleSlot` остаётся для хоткеев 1–6 — он становится тонкой обёрткой над `Install`.

## 4. Слайсинг

- **C1 — модель.** Человек как вариант, прямой `Install`/`Remove`, человеческие органы в химерном слоте, `Catalog()`. Старый список слотов продолжает работать на хоткеях и цикле. *Играбельно: химерный слот чинится сразу, до всякого нового UI.*
- **C2 — экран.** Фигура с гнёздами, созвездия доноров, drag&drop, подсветка совместимых, красный отказ по пулу, полная пауза. Список строк умирает. *Играбельно: сборка перестаёт быть циклом.*
- **C3 — красота.** Контур Витрувия, реальные рисунки созвездий, морф фигуры по свапам, яркость созвездия по родству. *Косметика поверх работающего.*

## 5. Открытые вопросы

- **Совместимость при драге.** Подсвечивать только гнездо родного типа слота (Коготь → Руки) или все, куда орган в принципе влезает (химерные гнёзда — куда угодно)? Предложение: родное гнездо + все свободные химерные.
- **Куда девать созвездия при росте бестиария.** Три донора по бокам помещаются; ёж и сова сделают пять. Прокрутка, вкладки по видам или карта с панорамированием? Решаем в C3, когда появится пятый вид, — раньше это гадание.
- ~~**Дубли в химерном слоте.**~~ → **решено (2026-07-21): один экземпляр органа на тело.** Состав химерного слота динамичен: надетое в другом слоте выпадает из доступных (список вариантов при этом стабилен — индексы не едут, иначе drag&drop ловил бы съехавшие). Дубль ДРУГОГО вида в тот же тип слота разрешён и осмыслен: волчья Пасть (вой) + змеиные клыки (яд) = обе фичи, супремум схлопывает только величины.

## 6. За горизонтом: смена структуры тела

Правило «один экземпляр органа» — ограничение **нынешней** модели, где набор слотов задан шасси. Замысел пользователя (2026-07-21, далеко за MVP): **мифические слоты** — менять саму структуру тела, а не только начинку. Отрастить четвёртую руку, вторую голову, сплавить шасси. Тогда «два Когтя» станет законным: это не дубль в одном теле, а две разные руки.

Что это значит для сегодняшних решений: правило одного экземпляра живёт в `WornElsewhere` **одним местом** — когда дойдёт до мифических слотов, снимать его придётся там же, а не выковыривать из десяти проверок. Структура «слоты от шасси» при этом становится не константой, а стартовым состоянием.
